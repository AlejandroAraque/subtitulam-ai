# Resumen v1.5 para la memoria del TFM

> **Fechas:** 22-25 abril 2026
> **Commits implicados:** `f84f00c` → `ff896fb` (6 commits)
> **Hito:** transición de prototipo en memoria (v1.x) a sistema persistente con backend canónico (v1.5).
> **Objetivo de la versión:** sentar la **infraestructura de datos y la trazabilidad** sin las cuales no es posible aplicar RAG en v2 ni medir su impacto cuantitativamente.

---

## 1. Contexto y motivación

Las versiones 1.0 y 1.1 (commits anteriores a `f84f00c`) implementaban un MVP funcional de traducción EN→ES con GPT-4o pero presentaban dos limitaciones bloqueantes para la siguiente fase:

1. **Ausencia de persistencia.** Glosario e historial residían en `st.session_state` de Streamlit, perdiéndose al recargar el navegador. No existía manera de recuperar ni auditar las traducciones realizadas.
2. **Métricas inventadas.** El frontend mostraba indicadores hardcodeados (98% CPL, 1.284 líneas) que no reflejaban el resultado real del backend, imposibilitando cualquier análisis empírico.

La versión 1.5 aborda exclusivamente estos dos problemas. Es deliberadamente una versión **infraestructural**: no introduce funcionalidad nueva visible al usuario, pero sin ella el RAG planteado para v2 no tendría dónde almacenar la memoria de traducciones previas que constituye su materia prima.

---

## 2. Decisiones de diseño y justificación

### 2.1 SQLite frente a alternativas

Se eligió **SQLite** sobre PostgreSQL/MySQL/MongoDB tras evaluar las siguientes restricciones:

| Criterio | Peso | Decisión |
|---|---|---|
| Concurrencia esperada (<10 usuarios simultáneos) | alta | SQLite suficiente |
| Coste de infraestructura | alta | SQLite = cero (archivo local) |
| Reproducibilidad (incluir BBDD en repos académicos) | alta | SQLite = un solo archivo, copiable |
| Posibilidad de migración futura | media | SQLAlchemy abstrae el dialecto, migrar es trivial |

La arquitectura mediante SQLAlchemy 2.x permite una migración futura a PostgreSQL con cambios mínimos en la URL de conexión, conservando la integridad del modelo.

### 2.2 SQLAlchemy 2.x con sintaxis declarativa

Frente al ORM 1.x clásico se adoptó la nueva API tipada (`DeclarativeBase`, `Mapped[T]`, `mapped_column`). Justificación:

- Mejor integración con type checkers (mypy, Pylance)
- API estable de cara al futuro (la 1.x está marcada como legacy)
- Mejor documentación oficial en 2026

### 2.3 Arquitectura por capas

Se aplicó el principio de **separación de responsabilidades**:

```
┌─────────────────┐
│   Presentation  │  app_ui.py  (Streamlit)
├─────────────────┤
│   Application   │  app/api/routes.py  (FastAPI)
├─────────────────┤
│     Domain      │  app/services/  (lógica de negocio)
├─────────────────┤
│      Data       │  app/models/schemas.py  (ORM)
├─────────────────┤
│ Infrastructure  │  app/core/{config,database}.py
└─────────────────┘
```

Cada capa solo conoce las inferiores. Los endpoints REST nunca instancian el ORM directamente — pasan por la capa de servicios. Los servicios reciben la sesión de BBDD como parámetro (Dependency Injection vía `Depends(get_db)` de FastAPI), evitando dependencias globales y facilitando la testabilidad.

## 3. Modelo de datos

Se diseñaron tres entidades con la relación que muestra el diagrama:

```
┌──────────────────┐       ┌─────────────────────────┐       ┌──────────────────┐
│  GlossaryTerm    │       │          Job            │       │   Translation    │
│──────────────────│       │─────────────────────────│       │──────────────────│
│  id              │       │  id                     │  1─N  │  id              │
│  source          │       │  filename               │ ────► │  job_id (FK)     │
│  target          │       │  target_lang            │       │  cue_idx         │
│  category        │       │  cpl                    │       │  source_text     │
│  note            │       │  context                │       │  target_text     │
│  created_at      │       │  started_at             │       │  created_at      │
└──────────────────┘       │  elapsed_s              │       └──────────────────┘
                           │  status                 │
                           │  cpl_compliance         │
                           │  tokens_prompt          │
                           │  tokens_completion      │
                           │  error                  │
                           └─────────────────────────┘
```

### Decisión clave: granularidad cue por cue

La tabla `Translation` almacena **una fila por subtítulo** (cue) en lugar del SRT completo. Esta atomización es deliberada: constituye la materia prima del sistema RAG previsto para v2, donde cada cue se indexará como vector en ChromaDB para búsqueda semántica de traducciones similares en jobs anteriores.

La relación 1-N entre `Job` y `Translation` se configuró con `cascade="all, delete-orphan"`, garantizando integridad referencial: borrar un job arrastra automáticamente sus translations.

## 4. API REST

Se expusieron 6 endpoints HTTP (1 modificado + 5 nuevos):

| Método | Ruta | Función |
|---|---|---|
| `POST` | `/translate` | (modificado) traduce + calcula métricas + persiste job atómicamente |
| `GET` | `/glossary` | lista términos del glosario |
| `POST` | `/glossary` | crea término (validado con Pydantic) |
| `DELETE` | `/glossary/{id}` | elimina término |
| `GET` | `/jobs` | lista historial de traducciones |
| `GET` | `/jobs/{id}` | devuelve job con todas sus translations |
| `DELETE` | `/jobs/{id}` | elimina job y sus translations en cascada |

### Detalles relevantes

- **Validación de entrada:** se utiliza `pydantic.BaseModel` con `Field(..., min_length=1)` para campos obligatorios. Los errores 422 incluyen estructura semántica (`type`, `loc`, `msg`, `input`) consultable por el cliente.
- **Metadata vía headers HTTP:** la respuesta de `/translate` incluye `X-Cpl-Compliance`, `X-Tokens-Prompt`, `X-Tokens-Completion`, `X-Elapsed-Seconds` para que el frontend obtenga métricas sin parsear el cuerpo binario.
- **Lifespan asyncio en FastAPI:** la inicialización de la BBDD (`init_db()`, idempotente) se invoca desde un context manager `lifespan` registrado en el constructor de `FastAPI`, sustituyendo a los deprecados `@app.on_event("startup")`.

## 5. Frontend

`app_ui.py` se refactorizó para tratar `st.session_state` como **caché de sesión**, no como almacén autoritativo. La fuente de verdad pasa a ser SQLite a través de la API REST.

### Patrón de helpers HTTP

Se introdujeron 5 funciones cortas que encapsulan las llamadas al backend con manejo defensivo de errores:

```python
def api_get_glossary() -> list[dict]:
    try:
        r = requests.get(f"{BACKEND_URL}/glossary", timeout=10)
        r.raise_for_status()
        return r.json()
    except requests.exceptions.RequestException:
        return []  # fallback degradado: la UI no explota
```

Este patrón "fail-soft" permite que la interfaz siga renderizándose si el backend está caído, mostrando estado vacío en lugar de una excepción no controlada.

### Renombrado de campos en el render

Los campos del API en inglés snake_case (`source`, `target`, `category`) se renombran a etiquetas en español (`Término`, `Traducción`, `Etiqueta`) en el momento de construir el `pandas.DataFrame`, manteniendo la separación entre el **modelo de datos** (estable, en inglés) y la **presentación** (localizable).

## 7. Test-set de evaluación

Se preparó un dataset inicial de 20 pares `(inglés, español_referencia_humana)` en formato JSONL (`data/testset/reference_en_es.jsonl`).

### Estructura del registro

```json
{"id": 1, "source": "...", "target": "...", "notes": "..."}
```

El campo `notes` documenta el tipo de dificultad lingüística de cada par (idiom, polisemia, jerga técnica, sarcasmo, formato HTML, etc.), permitiendo análisis estratificado de errores.

### Cobertura de dificultades

El test-set abarca 16 categorías de fenómenos lingüísticos relevantes para subtitulación profesional:

- Idioms culturales (5 cues): "in ages", "piece of cake", "raining cats and dogs", "cold feet", "had your back"
- Idioms especializados (3): "break a leg" (teatro), "Your Honor" (jurídico), "stat!" (médico)
- Polisemia con desambiguación contextual (3): "bank", "light"
- Jerga técnica (1): "server/firewall"
- Nombres propios y marcas (2): "Starbucks", "Penn Station", "Mr. O'Brien"
- Formato HTML inline (1): preservación de etiquetas `<i>...</i>`
- Sarcasmo, doble sentido, ambigüedad de género, multilínea con corte CPL (resto)

### Justificación del formato JSONL

Se eligió JSONL frente a JSON tradicional por tres razones documentables:

1. **Append-only friendly:** ampliar el dataset no requiere reparsear ni reescribir el archivo existente.
2. **Streamable:** procesable línea a línea sin cargar todo en memoria — esencial cuando se escale a 200-500 pares en v2.
3. **Estándar de facto en NLP:** OpenAI fine-tuning, Hugging Face Datasets, BigQuery ML utilizan JSONL como formato canónico.

## 8. Baseline canónico — resultados empíricos

Se ejecutó la traducción del SRT de prueba (`data/test_sample_en.srt`, 20 cues) con configuración baseline pura: contexto vacío, glosario no aplicado, sin RAG, modelo `gpt-4o`, `temperature=0.3`, `chunk_size=5`. Los artefactos del run quedan inmutablemente versionados en `data/baseline_run/`.

### Métricas registradas

| Métrica | Valor |
|---|---|
| Latencia total | 9,77 segundos |
| Tokens de entrada (prompt) | 1.787 |
| Tokens de salida (completion) | 305 |
| Tokens totales | 2.092 |
| Coste estimado (precios GPT-4o abril 2026) | ~$0,0083 USD |
| Cumplimiento CPL (líneas ≤ 42) | 100,0 % |
| Coincidencia exacta con referencia humana | 6/20 (30,0 %) |

### Análisis cualitativo

La cifra de 30 % de coincidencia exacta es **intencionadamente estricta** y debe interpretarse con cautela: muchas divergencias son variaciones léxicas legítimas (*"Te veré"* vs *"Te veo"*) o diferencias de formato (saltos de línea distintos) sin pérdida semántica. Esta limitación motiva la introducción en v2.1 de métricas semánticas (BLEU, BERTScore, chrF) que sí capturan la equivalencia funcional entre paráfrasis.

### Errores reales detectados (no léxicos)

Dos cues presentan defectos genuinos en el output de v1.5 que documentan los límites del enfoque sin RAG:

- **Cue 5 (polisemia):** *"river bank"* se traduce como *"banco del río"*. La traducción no es errónea estrictamente pero es subóptima — *"orilla del río"* es la opción natural en español. El sistema carece de contexto suficiente para preferirla. **Caso paradigmático que el RAG debe resolver consultando traducciones previas similares.**
- **Cue 14 (registro formal):** *"Mr. O'Brien"* mantiene la fórmula inglesa sin antepóner el artículo *"el"*. En registro jurídico español el artículo es obligatorio.

### Confirmación de fixes anteriores

El cue 15 — *"I `<i>`knew`</i>` you'd come through"* — preserva correctamente las etiquetas `<i>...</i>` en el output, validando empíricamente la corrección introducida en v1.1 (commit `8cc755c`) que añadía instrucciones explícitas de preservación de formato HTML inline al system prompt.

## 9. Limitaciones reconocidas y trabajo pendiente

Se identificaron las siguientes limitaciones de v1.5 que motivan las versiones siguientes:

| Limitación | Impacto | Versión que lo aborda |
|---|---|---|
| Cada chunk de 5 cues se traduce en aislamiento | Pérdida de coherencia entre cues contiguas (ejemplo cue 5) | v2.3 (RAG retrieval) |
| El glosario se almacena pero no se inyecta al LLM | Términos definidos no surten efecto en la traducción | v2.4 (glossary injection) |
| Métrica de evaluación basada en coincidencia exacta | Penaliza paráfrasis válidas | v2.1 (módulo de evaluación con BLEU/BERTScore/chrF) |
| Variedad de español no parametrizada (ES vs LATAM) | Riesgo dialectal según el corpus de entrenamiento de GPT-4o | v2.5 (parametrización completa) |
| Test-set de tamaño insuficiente (20 pares) | Baja potencia estadística | Ampliación a 200-500 pares en v2 con OpenSubtitles + TED2020 |
| Corte de líneas por longitud sin conciencia sintáctica | Cortes en posiciones gramaticalmente subóptimas | v2.5 (CPL syntax-aware) |

## 10. Roadmap a v2

```
v2.1   Módulo de evaluación con BLEU/BERTScore/chrF + adherencia al glosario
v2.2   Servicio de embeddings (OpenAI text-embedding-3-small) + ChromaDB
v2.3   Retrieval en /translate: top-K traducciones similares por cosine
v2.4   Inyección de glosario al prompt con matching por lema (spaCy)
v2.5   Repetición del baseline run con cada incremento → tabla de ablación
```

La tabla de ablación final (v2.5) constituirá el resultado central a presentar en la defensa: una comparación cuantitativa **baseline → +contexto → +glosario → +RAG → sistema completo** sobre el test-set ampliado.
